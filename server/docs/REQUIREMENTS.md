# 기능 요구사항

## API 설계

- endpoint와 HTTP 메서드를 REST 원칙에 맞게 스스로 설계한다
- HTTP 상태 코드(200, 204, 201, 400, 404, 500 등) 사용 기준을 정하고 그 이유를 설명할 수 있다

### 상품

- [ ] 상품은 고유 id, 이름, 가격, 이미지를 가진다.
- [ ] GET 요청의 endpoint는 `/products`이고, 성공 status는 200이다.
- [ ] POST 요청의 endpoint는 `/products`이고, 성공 status는 201이다.
- [ ] DELETE 요청의 endpoint는 `/products/:id`이고, 성공 status는 204이다.

### 장바구니

- [ ] 상품 id, 수량을 가진다.
- [ ] GET 요청 시, endpoint는 `/carts`이고, 성공 status는 200을 가진다.
- [ ] PATCH 요청 시, endpoint는 `/carts/:id`이고, 성공 status는 204를 가진다.
- [ ] DELETE 요청 시, endpoint는 `/carts/:id`이고, 성공 status 204를 가진다.

## 서버

- Node.js + Express를 사용한다.
- TypeScript 로 구현한다.
- 데이터는 Map으로 관리한다.
- HTTP 상태 코드를 의미에 맞게 사용한다 (200, 201, 400, 404 등)
- FE와 연동을 위한 CORS 설정을 한다.

### 도메인 테스트 시나리오

#### 상품

##### 정상 동작

1. 상품명은 1자 ~ 100자이다.
2. 가격은 0보다 큰 숫자이다.

##### 예외 처리

1. 상품명이 공백이면 예외 처리한다.
2. 상품명이 100자 초과이면 예외 처리한다.
3. 가격이 0 이하이면 예외 처리한다.
4. 가격이 숫자가 아니면 예외 처리한다.
5. 중복된 상품명이면 예외 처리한다.

#### 장바구니

##### 정상 동작

1. 수량은 1 이상 99 이하 정수이다.

##### 예외 처리

1. 수량이 0 이하이면 예외 처리한다.
2. 수량이 100 이상이면 예외 처리한다.

### API 호출 테스트 시나리오

#### 상품

##### POST

###### 정상

- POST 요청 시 상품을 추가한다.

###### 예외

1. 상품명이 공백이면 400을 응답한다.
2. 상품명이 100자 초과이면 400을 응답한다.
3. 가격이 0 이하면 400을 응답한다.
4. 가격이 숫자가 아니면 400을 응답한다.
5. 중복된 상품명이면 400을 응답한다.
6. 요청시간이 15초를 넘기면 408을 응답한다.
7. 네트워크 에러 및 알 수 없는 에러는 500을 응답한다.
8. 서버에서 기능 구현이 되지 않은 경우 501을 응답한다.

##### GET

###### 정상

- GET 요청시 상품 목록을 반환한다.

###### 예외

1. 요청 url이 잘못된 경로인 경우 404를 응답한다.
2. 요청시간이 15초를 넘기면 408을 응답한다.
3. 네트워크 에러 및 알 수 없는 에러는 500을 응답한다.
4. 서버에서 기능 구현이 되지 않은 경우 501을 응답한다.

##### DELETE

- DELETE 요청 시 상품을 삭제한다.
- DELETE 요청 시 장바구니에서 상품을 삭제한다.

###### 예외

1. 요청 url이 잘못된 경로인 경우 404를 응답한다.
2. 요청시간이 15초를 넘기면 408을 응답한다.
3. 네트워크 에러 및 알 수 없는 에러는 500을 응답한다.
4. 서버에서 기능 구현이 되지 않은 경우 501을 응답한다.

- 존재하지 않는 상품/장바구니 아이템 요청 시 에러를 응답한다
- quantity는 1 이상 99 이하의 정수여야 한다
- price는 0보다 큰 숫자여야 한다
- 상품명은 최대 100자이다
